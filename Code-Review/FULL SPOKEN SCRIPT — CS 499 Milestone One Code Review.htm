Category One: Software Engineering & Design — CS‑410 Reverse‑Engineered C++ Program

(1:00–12:00)

Existing Functionality (1:00–3:00)

“The first artifact is a C++ program I reconstructed in CS 410 as part of a reverse‑engineering proficiency test. The original binary implemented a simple client‑management menu system, and my task was to recreate the source code while identifying vulnerabilities.

The program displays a menu with three options: change a customer choice, display customer information, or exit. It uses a global state variable, several helper setup routines, and functions that simulate customer‑information handling. The program accepts user input, updates the global state, and executes the corresponding action.”

Code Analysis Using the Checklist (3:00–10:00)

Structure

“When reviewing the structure, I found several issues:

  • The program relies on a global mutable state variable, which is risky in larger systems.
  • All logic is contained in a single file with no modular separation.
  • The helper functions are placeholders with no meaningful implementation.
  • Menu logic, permission logic, and customer‑information logic are tightly coupled.
  • Hardcoded values appear throughout the code.
  • There is no abstraction, no encapsulation, and no separation of concerns.

Overall, the structure works for a small demonstration but is not scalable or maintainable.”

Documentation

“The code includes minimal comments and no method‑level documentation. There is no explanation of the program’s purpose, expected inputs, or behavior. This makes the code harder to understand and maintain.”

Variables

“Variable usage is inconsistent:

  • The global state variable is used for multiple purposes.
  • Temporary variables like input and choice are not validated beyond basic type checking.
  • Hardcoded values appear throughout the code.
  • No type hints or descriptive naming conventions are used.

These issues reduce readability and increase the risk of logic errors.”

Arithmetic Operations

“Arithmetic operations are minimal, but the vulnerabilities come from lack of validation. For example, the program does not enforce numeric ranges, and invalid input can cause unexpected behavior. There are no safeguards against malformed input.”

Loops and Branches

“The main loop runs indefinitely until the user selects exit. Issues include:

  • No explicit termination condition beyond user input.
  • No default case for unexpected state values.
  • Branching logic is tied directly to the global state.
  • Some logic inside the loop could be moved outside to improve clarity.

These issues make the control flow harder to maintain.”

Defensive Programming

“This is where the most significant vulnerabilities appear:

  • No exception handling for input operations.
  • No validation of user permissions — the permission function always denies access.
  • Hardcoded sensitive data in the display function.
  • No sanitization of user input.
  • No error handling for invalid states.
  • Unsafe global state manipulation.

These issues would be unacceptable in a production environment.”

Planned Enhancements (10:00–12:00)

“To address these issues, I plan to:

  • Replace the global state with a MenuController class.
  • Move customer‑information logic into a dedicated class.
  • Replace hardcoded values with configuration constants.
  • Add full input validation and structured exception handling.
  • Implement role‑based access control.
  • Add documentation and a README.
  • Introduce unit tests for menu navigation and permission logic.

These enhancements demonstrate mastery in modular design, defensive programming, maintainability, and secure coding.”

Category Two: Algorithms & Data Structures — CS‑370 Q‑Learning Agent

(12:00–22:00)

Existing Functionality (12:00–14:00)

“The second artifact is my Q‑learning agent from CS 370. This program trains an agent to navigate a grid‑based maze using reinforcement learning.

The agent uses a neural‑network‑based Q‑value predictor, updates Q‑values based on rewards, and runs episodes until it converges on an optimal policy. It uses exploration versus exploitation logic, experience replay, and a decaying learning rate.”

Code Analysis Using the Checklist (14:00–20:00)

Structure

“The structure is very flat. The environment, the agent logic, and the training loop are all mixed together in one function. There are no classes, no abstraction, and no separation of concerns. This makes the code harder to extend or reuse.”

Documentation

“Documentation is minimal. There are a few comments, but no docstrings. Hyperparameters like learning rate, decay rate, and batch size are not explained.”

Variables

“Variable naming is inconsistent. The grid is hardcoded. There are no type hints. Some variables inside the training loop are created but never used or overwritten.”

Arithmetic Operations

“The Q‑value update logic is correct, but there are no safeguards against floating‑point issues. There are no checks for invalid reward values or invalid state transitions.”

Loops and Branches

“The episode loop does not have a clear or configurable termination condition. There is no default case for invalid actions. Some logic inside the loop could be moved outside to improve performance.”

Defensive Programming

“There is no validation of environment size or state transitions. There is no bounds checking on actions. There is no error handling for invalid states or rewards. The program assumes everything is valid, which is risky.”

Planned Enhancements (20:00–22:00)

“I plan to restructure the entire artifact using object‑oriented design. I’ll create separate classes for the environment, the agent, and the state representation.

I’ll convert the Q‑table into a dictionary keyed by state‑action pairs, which is more flexible and scalable. I’ll add configurable hyperparameters, logging for training episodes, and unit tests for Q‑value updates and convergence.

These enhancements demonstrate mastery in algorithms, data structures, abstraction, and testing.”

Category Three: Databases — CS‑340 MongoDB CRUD Class

(22:00–29:00)

Existing Functionality (22:00–24:00)

“The third artifact is a Python class that connects to MongoDB and performs CRUD operations on the AAC database. It uses PyMongo for database access and exposes methods for create, read, update, and delete.

The class accepts dynamic authentication credentials and interacts with the animals collection.”

Code Analysis Using the Checklist (24:00–28:00)

Structure

“The biggest issue is that database logic is mixed directly into the class with no abstraction layer. There is no schema validation, no indexing strategy, and no reusable query helpers. Hardcoded host and port values also reduce flexibility.”

Documentation

“Documentation is minimal. There is a single class‑level docstring, but no method‑level documentation and no explanation of the expected schema.”

Variables

“Variable naming is inconsistent. There are no type hints. Some variables are generic, like data, criteria, and result. Hardcoded values appear throughout the code.”

Arithmetic Operations

“Not heavily used, but numeric fields are not validated. There are no checks for invalid numeric input.”

Loops and Branches

“Some operations assume that documents exist and do not handle missing results gracefully. There is no default handling for invalid criteria.”

Defensive Programming

“This is where the most serious issues appear:

  • No input sanitization
  • No schema validation
  • No parameterized queries
  • No try/except around most operations
  • No logging
  • No connection management
  • No graceful handling of missing documents

These issues pose security and reliability risks.”

Planned Enhancements (28:00–29:00)

“I plan to add schema validation using Pydantic or custom validation logic. I’ll add indexes for frequently queried fields. I’ll create a dedicated data access layer to separate database logic from application logic.

I’ll add parameterized queries, error handling, and logging. I’ll add docstrings, comments, and unit tests using a mock MongoDB instance.

These enhancements demonstrate mastery in database design, security, software engineering, and testing.”
